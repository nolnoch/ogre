55 /* Since we should not have the physics sim running on clients, the server needs
56  * to keep track of and distribute all ball locations and velocities. This
57  * struct theoretically can handle up to 27 balls at one time by using 64 bits
58  * per ball and keeping track of how many balls/64-bit fields should be read.
59  *
60  * Per ball:
61  * 11 bits - x position interpreted as integer (max 2048)
62  * 11 bits - y position interpreted as integer (max 2048)
63  * 11 bits - z position interpreted as integer (max 2048)
64  * 10 bits - x velocity interpreted as integer (max 1024)
65  * 10 bits - y velocity interpreted as integer (max 1024)
66  * 10 bits - z velocity interpreted as integer (max 1024)
67  * 1 bit - padding
68  *
69  * This is very low-level and slightly hacky, but it could allow us to update
70  * all balls with only 220 bytes every single network update. It is possible
71  * to update all balls with full Vector3s, but it would require 652 bytes per
72  * update (for 27 balls).
73  *
74  * The current network buffer size is 128 bytes. This can change but must be
75  * lower than the MTU (maximum transmission unit) set by the hardware and the
76  * network. This is commonly 1500 bytes but can be as low as 500. Size also
77  * slows down the transmission. We can discuss and/or test.
78  */
